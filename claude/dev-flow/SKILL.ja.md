# dev-flow — 段階開発フロー

[English](SKILL.md) | **日本語**(参考訳 — 実行時に読み込まれる正本は英語版)

> 開発を8つの工程 — 構想 → 事前調査(scout) → 設計 → PoC → 確証 → 実装 →
> 手動での本番反映 → 観察 → 確定 — に分けて進めるための規律。ある程度の規模の
> feature・設計・issue 作業を始めるとき、「いまどの工程?」を確認したいとき、
> 次の工程へ進んでいいか判定したいとき、そして本番を変更する前に使う。
> 案件ごとの進行管理ファイル、fail-closed な通過条件(gate)、証拠(receipt)の
> 雛形を提供する。

## 原則(一行で)

**現実に触れるタイミングを、できるだけ安い工程まで前倒しする。**

高くつく工程 — 本番の変更、長時間のパイプライン、承認の消費 — は「うまくいくと
分かっていることの確認」に使う場所であって、「うまくいくか試す」場所ではない。
各工程は前の工程の receipt を入力にとる。これによって、頭の中の環境イメージだけで
設計が完結してしまうことを防ぐ。

このフローは実際の失敗から生まれた。想像上の環境を前提に書いた設計が、最初に
実態と突き合わせた瞬間に崩れた。1時間かかるリリースパイプラインを「動くか試す」
ために何度も回した。手元の安価な検証で見つかったはずの欠陥に、本番の一発勝負を
消費した — どれも「現実に触れるのが遅すぎた」事故だ。

## 8つの工程

| # | 工程 | 成果物(receipt) | 次へ進む条件(gate) |
|---|---|---|---|
| 0 | 構想メモ | 何を・なぜ・成功条件を1枚(これは設計ではない — scout にどこを見てもらうかの指定) | — |
| 1 | 事前調査(scout) | 実行コマンドを添えた読み取り専用の実態調査。調べきれなかった箇所も正直に記録 | receipt が封緘済みで、設計が拠って立つ事実すべてに裏付けがある |
| 2 | 設計 | scout の実測に基づいた設計文書 + 独立レビューの判定 | blocking 指摘ゼロ。scout の欠測が「後続の blocking」として明記されている |
| 3 | PoC | 使い捨ての検証結果: 何を証明したか、**そして何を証明していないか** | 設計中の「初めて試す要素」すべてに白黒がついている |
| 4 | 確証 | 実行手順全段の事前検証ログ + 封印したパケット | 実行手順の全段が隔離環境で green — 「本番で初めて試すことは何もない」と証明できた |
| 5 | 実装 | マージ済み PR(テスト・CI・レビュー込み) | CI green + レビュー承認 + 設計との整合 |
| 6 | 手動での本番反映 | 消えない実行記録(記録が先、実行が後) | 工程4のパケットが有効で、封印後に実行手順へ変更が入っていない |
| 7 | 観察 | 反映後の実測値(readback)と経過観察の記録 | 実測が期待と一致し、観察期間を走りきった |
| 8 | 確定 | 封緘した証拠、issue のクローズ、教訓の書き戻し | — |

工程ごとの絶対ルール:

- **scout は読み取り専用を厳守。**「`<コマンド>` で確認した」と書く。「〜のはず」は
  禁止。権限不足(403)や到達不能は「未確認 = 後続で blocking」と記録し、推測で
  埋めない
- **PoC の成果物は使い捨て。** PoC のコードや fixture を本番に流用しない
- **本番反映は手動・一発勝負・記録が先。** 実行前に試行の記録を残し、1回だけ実行し、
  結果を消えない形で残す。自動リトライも自動ロールバックもしない
- **観測手段は設計(工程2)の時点で織り込む。** 本番に出してから慌てて監視を
  作るのは、運用の課題ではなく設計の欠陥

## hollow-green への防御(工程5)

worker は skeleton 実装でも test を green にできてしまう。「green」は品質の
代理指標にすぎず、安いモデルほど代理指標を文字どおり最適化する(本番運用で
繰り返し捕捉した実話)。防御はどれも実戦で鍛えたもの:

- **test の所有権を分ける(contract-test-first)。** RED test は実装とは別の、
  より強い lane が「production の経路で実際に落ちる」形で書いて所有する。
  実装 lane の仕事はそれを green にすることだけ。**実装 lane が test に diff を
  当てたら、その場で差し戻し**
- **本物の境界での green だけを gate にする。** fake-client や純粋 unit test の
  green は工程3(PoC)の証拠止まり。工程5 の merge gate は本物の境界を持つ
  harness(実 API サーバなど)での green を要求する。偽物の環境では、誠実な
  実装と空っぽの実装の区別が原理的につかない — 真面目さの問題ではなく構造の限界
- **held-out test。** レビュー時に、実装者が一度も見ていない test を1本追加して
  走らせる。skeleton は必ず落ち、仕様を実装したものは通る。レビュー手順に
  1行足すだけで、いちばん安くて強い検出器になる
- **同じ lane の hollow-green 2回目は遮断イベント。** そこで止めて、やり方を
  変える — contract-test-first への転換、台帳管理のブースト、issue の切り直し。
  これは怠慢ではなく「scope が大きすぎる・gate が壊れている・モデルの力が
  足りない」のシグナルだ
- **gate は正直に通れるものであること。** test gate を設計レビュー(工程2)する
  ときは「誠実な実装がこの環境で本当に green になれるか」を確認する。正直では
  通れない gate は、捏造を唯一の通り道にしてしまう

## 進行管理ファイル

案件ごとに `flow/<topic>-flow.md` を1本作る(作業ディレクトリ配下)。
この案件がいまどの工程にいるかの、唯一の正式な記録になる。

```markdown
# flow: <topic>
- 開始: <日付> / 担当: <lane または人>
- 現在の工程: <N>. <名前>

| 工程 | 状態 | receipt(パス + SHA) | 備考 |
|---|---|---|---|
(0〜8 の行)

## 差し戻し履歴
- <日付> 工程N → M: <理由を1行>
```

工程を進めるたび・戻すたびに必ず更新する。receipt 欄は**パス + SHA** で埋める。
口頭の「終わりました」は、終わったことにならない。

## 差し戻し先の早見表

| ズレが見つかった場所 | 戻る工程 |
|---|---|
| 設計レビューで、scout にない前提が見つかった | 1(調査のやり直し、または設計の修正)|
| PoC で設計の仮説が崩れた | 2 |
| 確証で、未検証の段が残っていた | 3 |
| 実装中に設計の前提とのズレに気づいた | 2(設計を改版する — その場のつじつま合わせは禁止)|
| 本番反映が失敗した(実行記録は残っている) | 4(パケットを再発行し、方式を点検)|
| 観察で問題が見つかった | 状況判断(2・5・6 のいずれか)— 自動ロールバックはしない |

**gate を満たしていないのに進みたくなったら**: 間違っているのは gate ではなく、
上流のどこかだ。進む方法ではなく、戻る先を探すこと。

## 誰が・どのモデルでやるか(worker fleet を使う場合)

- 実装系の工程(3〜5、7)は **fleet の標準モデル**の worker に任せる
  (例: Codex fleet なら `gpt-5.6-terra` medium、Claude Code fleet なら **`sonnet`**)
- 上位モデル(Codex `sol` は effort low が基本 / Claude Code `opus`)は
  「行き詰まりの突破」と「重要局面のスポットレビュー」専用。**必ず台帳に記載**し、
  常駐させない
- 工程6(本番反映)はモデルではなく**人間の関門**: オーナー本人か、明示的に
  権限を委ねられた実行者だけが行う
- lane の作り方と寿命は issue-lane スキル(1 issue = 1 lane)に従う

## 停止条件(暴走の遮断器)

進捗は成果 — 実際の変更、反映後の実測、残り工程 — で数える。receipt の枚数や
後続チケットの数ではない。同じ操作で2回却下された、30分停滞した、手順の不備が
3連続した — どれかに達したら次の試行を止め、方式そのものを設計し直す。
fail-closed は「壊れたやり方を安全に無限に繰り返してよい」という意味ではない。

## プロジェクトへの導入

プロジェクトに正式なフロー文書がすでにあるなら、そちらが優先。このスキルは
正本を持たないプロジェクトのための既定値だ。導入するときは、上の工程表を
プロジェクトの正本文書としてコピーし、そのプロジェクト固有の gate・担当・
パス規約をそこに書き足していく。

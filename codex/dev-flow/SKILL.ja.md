# dev-flow — 軽量開発フロー(既定トラック)

[English](SKILL.md) | **日本語**(参考訳 — 実行時に読み込まれる正本は英語版)

> 日常の開発作業のための既定トラック。source-only の変更、dev/検証環境への
> live 反映、実利用者ゼロの本番(オーナー宣言つき)が対象。証拠はマージ済み
> PR + green CI + 台帳一行だけ — 封緘なし、successor 起票なし、verdict 文書の
> 往復なし。不可逆・顧客可視の作業は1問のトラック判定で prod-flow へ振り分ける。
> hollow-green 対策と「live で見つけた問題は再発防止 gate つきで close」の
> 規律を含む。

## 原則(一行で)

**厳密さは不可逆な mutation gate に集中させる。それ以外は速く出して、
CI + PR レビュー + regression gate に規律を担わせる。**

実行時の fail-closed チェックと同じ内容を事前の儀式で証明するのは、同じ安全に
二度払いしているだけだ。

この分割は実際の失敗から生まれた。一発勝負の不可逆な「発射」のために作られた
フロー(封緘 receipt・successor 起票・verdict 文書)が日常の開発にまで漏れ出し、
すべてを遅くした — しかも適用した先では安全を何も足していなかった。

## トラック判定(毎回、最初に問う)

**「これが失敗したら、返ってこない消費財(承認・署名・出版・quota)が減るか?
あるいは実顧客の目に触れるか?」**

- **No → このスキル(軽量、既定)。**
- **Yes → [prod-flow](../prod-flow/SKILL.md)** — packet 封緘と人間の GO gate を
  備えたフル段階フロー。

例:

- 軽量: source-only の変更(PR/CI で完結)、dev/検証環境への live 反映、
  実利用者ゼロの本番(オーナー宣言つき — 下記)。
- prod-flow: 顧客可視の本番 mutation、署名・出版(publication semantics)、
  組織権限(アカウント発行・権限変更)、一発勝負の tuple を消費するものすべて。

prod-flow の装備(packet・封緘・successor 起票・write-ahead receipt)を
軽量トラックへ持ち込まないこと。

## 軽量トラックのルール

- **封緘なし・receipt SHA 報告なし・successor 起票なし。** 失敗したら
  「直してもう一度実行する」だけ。
- 証拠 = マージ済み PR + green CI + プロジェクト台帳(CHANGELOG/EVENT 等)への
  追記一行。
- scout = read-only で見てきて、メモを issue に貼るだけ。read-only の規律は
  変わらない。見ずに推測で埋めるのも変わらず禁止。
- PoC と確証は統合 — 「隔離環境で1回動くのを見せる」程度。要否は実装者の判断。
- live 反映のあとは readback を一言残す(事実だけ、儀式なし)。
- 進行管理ファイルは不要。issue / PR がそのまま正となる。

## 削らないもの(儀式ではなく実効のある gate)

- **hollow-green 対策。** テスト所有権の分離(RED テストは別 lane が書いて所有し、
  実装者がテストに diff を入れたら即差し戻し)、レビュー時の held-out テスト、
  real-boundary green を merge gate に。偽物の環境は、正直な実装と空洞の実装を
  見分けられない。
- **1 issue = 1 lane**(issue-lane スキル)。
- **環境の識別は実測する。思い込みで判定しない** — 環境の取り違えは速度とは
  無関係の事故クラスであり、儀式を削ることはこれを削る免罪符にならない。

## 実利用者ゼロの本番(オーナー宣言つきの特例)

実利用者がいない本番は、オーナーの明示的な裁定によりこのトラックで回してよい。
引き換えの規律はただ1つ:

- **live で見つけた問題は、fix と再発防止 gate(regression テスト / QA walk /
  harness への追加)を同じ変更に入れてはじめて close できる。**
- 同じ問題が live に2回出たら gate の欠陥 → circuit breaker: 止めて gate を
  設計し直す。「もっとがんばる」で対応しない。
- 特例は実利用者が来た日に失効する。失効はオーナーが宣言する。勝手に延長
  しない。

これは「事前の儀式で守る」から「事後に構造で固める」への転換だ:
速くリリースし、問題は live で見つけ、見つけた問題は二度と起きない形にして返す。

## レビューのルール(verdict の往復をしない)

- このトラックでは CONFORMS / NOT-CONFORMS の verdict 文書を使わない。
  レビュー = PR の approve / request-changes + inline 指摘。
- レビュアーは **blocking 指摘を初回で全部出す**(小出し禁止)。初回に見えて
  いたはずの blocking を後出しするのはレビュアー側の欠陥(rework が新たに
  壊した箇所は別)。
- 再レビューは**指摘した変更の差分だけ**を確認する。
- **2往復で収束しなければ circuit breaker**: 非同期の往復を止め、レビュアーと
  実装者を同じ会話に入れるか、オーナーが残論点を裁定する。

## 停止条件(circuit breaker)

進捗は成果(マージ済み PR、readback)で数える。プロセスの産物では数えない。
同一 operation の 2 rejection、30分の停滞、同一 lane からの hollow-green
2回目のいずれかに達したら: 止めて方式を変える(issue を切り直す、
contract-test-first に切り替える、台帳制のギアブーストを使う)。

## プロジェクトへの適応

プロジェクトに固有の正本フロー文書があればそちらが優先。本スキルは正本が
無いプロジェクトの既定値。重い経路には [prod-flow](../prod-flow/SKILL.md) を
組み合わせ、トラック裁定(そのプロジェクトで何が消費財・顧客可視にあたるか)は
プロジェクトの正本文書に蓄積していく。

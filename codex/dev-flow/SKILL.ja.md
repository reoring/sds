# dev-flow — 段階開発フロー(Codex 版)

[English](SKILL.md) | **日本語**(参考訳 — 実行時に読み込まれる正本は英語版)

> 開発を8つの工程 — 構想 → 事前調査(scout) → 設計 → PoC → 確証 → 実装 →
> 手動での本番反映 → 観察 → 確定 — に分けて進めるための規律。ある程度の規模の
> feature・設計・issue 作業を始めるとき、自分のタスクがどの工程かを確認する
> とき、本番を変更する承認を求める前に使う。工程ごとの証拠(receipt)と
> fail-closed な通過条件(gate)を守らせる。

## 原則

**現実に触れるタイミングを、できるだけ安い工程まで前倒しする。**

高くつく工程 — 本番の変更、長時間のパイプライン、承認の消費 — は「うまくいくと
分かっていることの確認」に使う場所であって、「うまくいくか試す」場所ではない。
各工程は前の工程の receipt を入力にとるから、頭の中の環境イメージだけで設計が
完結することはできない。

## 8つの工程

| # | 工程 | 成果物(receipt) | 次へ進む条件(gate) |
|---|---|---|---|
| 0 | 構想メモ | 何を・なぜ・成功条件を1枚(設計ではない) | — |
| 1 | 事前調査(scout) | 実行コマンドを添えた読み取り専用の実態調査。調べきれなかった箇所も正直に | receipt 封緘済み。設計が拠って立つ事実すべてに裏付け |
| 2 | 設計 | scout の実測に基づく設計文書 + 独立レビュー | blocking 指摘ゼロ。scout の欠測が blocking として明記 |
| 3 | PoC | 使い捨ての検証結果: 何を証明したか、**何を証明していないか** | 「初めて試す要素」すべてに白黒がついている |
| 4 | 確証 | 実行手順全段の事前検証ログ、封印したパケット | 全段が隔離環境で green — 「本番で初めて試すことは何もない」と証明できた |
| 5 | 実装 | マージ済み PR(テスト・CI・レビュー込み) | CI green + 承認 + 設計との整合 |
| 6 | 手動での本番反映 | 消えない実行記録 | パケットが有効で、工程4以降に手順の変更なし |
| 7 | 観察 | 反映後の実測(readback)と経過観察の記録 | 実測が一致、観察期間を完走 |
| 8 | 確定 | 封緘した証拠、issue クローズ、教訓の書き戻し | — |

## worker lane が守ること

- **自分の工程を知る。** タスク指示にはどの工程の作業かが書いてあるはず。上流の
  receipt(scout の記録、設計レビューの判定、PoC の記録、パケット)が見当たら
  なければ、止まって報告する — 記憶から作り直さない
- **scout は読み取り専用を厳守。** 調査系コマンドだけ(`gh api` の GET、
  `aws describe|list|get`、`kubectl get`、ソースを読む)。事実は
  「`<コマンド>` で確認した」と書く。「〜のはず」は禁止。権限不足(403)や
  到達不能は「未確認 = 後続で blocking」と記録し、推測で埋めない
- **PoC の成果物は使い捨て。** PoC のコードや fixture を本番に流用しない
- **工程6は人間の関門。** 本番の変更は手動・一発勝負・記録が先(実行前に試行を
  記録 → 1回だけ実行 → 消えない記録を残す)。自動リトライも自動ロールバックも
  しない。封印済みの工程4パケットなしに本番を変えろと言われたら、**断って報告する**
- **待機する前に receipt を出す。** パス + SHA。receipt のない仕事は、
  なかったのと同じ

## 誰が・どのモデルでやるか(worker fleet を使う場合)

- 実装系の工程(3〜5、7)は fleet の標準モデルの worker に任せる — Codex lane は
  `gpt-5.6-terra` medium、Claude Code lane は **`sonnet`** が標準の実装担当
- 上位モデル(Codex `sol` は effort low が基本 / Claude Code `opus`)は台帳管理の
  ブーストとスポットレビュー専用 — 常駐させない
- 工程6(本番)は人間の関門: オーナーか、明示的に権限を委ねられた実行者。
  モデルではない

## 停止条件(暴走の遮断器)

進捗は成果 — 実際の変更、反映後の実測、残り工程 — で数える。receipt や後続
チケットの枚数ではない。同じ操作で2回却下、30分の停滞、手順不備の3連続 —
どれかに達したら後続の生成を止めて、方式そのものを設計し直す。fail-closed は
「壊れたやり方を安全に繰り返してよい」という意味ではない。

## プロジェクト正本の優先

repo や vault に正式なフロー文書があるなら、そちらが優先。このスキルは正本を
持たないプロジェクトのための既定値だ。新しいプロジェクトでは工程表を正本文書に
コピーし、固有の gate・担当・パス規約をそこに書き足していく。

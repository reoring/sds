# prod-flow — フル規律の段階フロー

[English](SKILL.md) | **日本語**(参考訳 — 実行時に読み込まれる正本は英語版)

> 不可逆・顧客可視の作業のためのフル規律フロー — 構想 → 事前調査(scout) →
> 設計 → PoC → 確証 → 実装 → 手動での本番反映 → 観察 → 確定。失敗すると
> 返ってこない消費財(承認・署名・出版・quota)が減るか、実顧客の目に触れる
> 作業に使う。packet 封緘(GO を packet digest に束縛)、工程ごとの receipt、
> fail-closed な gate、レビュー往復の上限、circuit breaker を提供する。
> 日常の作業には dev-flow を使うこと。

## 適用条件(と、適用しない条件)

まずトラック判定を問う:

**「これが失敗したら、返ってこない消費財(承認・署名・出版・quota)が減るか?
あるいは実顧客の目に触れるか?」**

- **Yes → このスキル。** 顧客可視の本番 mutation、署名・出版
  (publication semantics)、組織権限(アカウント発行・権限変更)、
  一発勝負の tuple を消費するものすべて。
- **No → [dev-flow](../dev-flow/SKILL.md)**(軽量、既定)。日常の作業に
  このフローを回すことが「プロセスが重い」事故の正体になる — 以下の装備は
  発射のためのものであって、反復のためのものではない。

## 原則(一行で)

**現実に触れるタイミングを、できるだけ安い工程まで前倒しする。**

高くつく工程 — 本番の変更、長時間のパイプライン、承認の消費 — は「うまくいくと
分かっていることの確認」に使う場所であって、「うまくいくか試す」場所ではない。
各工程は前の工程の receipt を入力にとる。これによって、頭の中の環境イメージだけで
設計が完結してしまうことを防ぐ。

このフローは実際の失敗から生まれた。想像上の環境を前提に書いた設計が、最初に
実態と突き合わせた瞬間に崩れた。1時間かかるリリースパイプラインを「動くか試す」
ために何度も回した。手元の安価な検証で見つかったはずの欠陥に、本番の一発勝負を
消費した。そして、曖昧な口頭承認のもとで「承認されたものと違うもの」が実行された。

## 8つの工程

| # | 工程 | 成果物(receipt) | 次へ進む条件(gate) |
|---|---|---|---|
| 0 | 構想メモ | 何を・なぜ・成功条件を1枚(これは設計ではない — scout にどこを見てもらうかの指定) | — |
| 1 | 事前調査(scout) | 実行コマンドを添えた読み取り専用の実態調査。調べきれなかった箇所も正直に記録 | receipt が封緘済みで、設計が拠って立つ事実すべてに裏付けがある |
| 2 | 設計 | scout の実測に基づいた設計文書 + 独立レビューの判定 | blocking 指摘ゼロ。scout の欠測が「後続の blocking」として明記されている |
| 3 | PoC | 使い捨ての検証結果: 何を証明したか、**そして何を証明していないか** | 設計中の「初めて試す要素」すべてに白黒がついている |
| 4 | 確証 | 実行手順全段の事前検証ログ + 封印した packet | 実行手順の全段が隔離環境で green — 「本番で初めて試すことは何もない」と証明できた |
| 5 | 実装 | マージ済み PR(テスト・CI・レビュー込み) | CI green + レビュー承認 + 設計との整合 |
| 6 | 手動での本番反映 | 消えない実行記録(記録が先、実行が後) | 工程4の packet が有効で、封印後に実行手順へ変更が入っていない |
| 7 | 観察 | 反映後の実測値(readback)と経過観察の記録 | 実測が期待と一致し、観察期間を走りきった |
| 8 | 確定 | 封緘した証拠、issue のクローズ、教訓の書き戻し | — |

工程ごとの絶対ルール:

- **scout は厳密に read-only。** 事実は「`<コマンド>` で実測した」と書く。
  「〜のはず」は書かない。403 や到達不能は「未観測 = 後続の blocking」として
  記録し、推測で埋めない。
- **PoC の成果物は使い捨て。** PoC のコードや fixture を本番に転用しない。
- **本番反映は手動・一発・記録先行**: 実行前に attempt receipt を書く →
  1回だけ実行 → 消えない記録を残す。自動リトライ・自動ロールバック禁止。
- **観測手段は工程2で設計に含める。** live 後に観測を後付けするのは設計の
  欠陥であって、運用のタスクではない。

## packet(工程4の成果物)

packet は**発射手順書であって、荷物そのものではない**: 何を・どこへ・どうやって
「ちょうど1回」実行するかを、判断の余地ゼロまで固定した文書だ。

- 中身はすべて数値で束縛する: source の exact SHA、成果物の全 inventory と
  digest、実行順序、「これ以外は触らない」の明示リスト、逸脱時の停止条件。
- 埋め終わると文書全体のハッシュ(**packetDigest**)が確定する。独立レビューも
  オーナーの GO も**その digest に対して**発行する — 「なんとなく承認したら
  違うものが実行された」が構造的に起きなくなる。
- 実行者は packet を読んでそのとおり1回だけ実行する。途中の「ついでにこれも
  直しとくか」は禁止。逸脱したら STOP して消えない記録を残し、工程4へ戻る
  (packet を再発行する)。

ロケット打ち上げの手順書だと思えばいい: 荷物は digest で特定済みで、紙には
「どの条件をすべて満たしたときだけ発射してよいか」が書いてある。

## レビュー往復の上限(verdict の往復をしない)

独立レビューは工程2と4で必須だが、往復には上限を課す:

- **blocking 指摘は初回で全部出す**(小出し禁止)。初回に見えていたはずの
  blocking を後出しするのはレビュアー側の欠陥(rework が新たに壊した箇所は別)。
- **再レビューは指摘した変更の差分だけ**を確認する — 毎回のフルレビューを
  やり直さない。exact-SHA の束縛は最終封緘の1回だけに適用する。
- **2往復で収束しなければ circuit breaker**: 非同期の往復を止め、レビュアーと
  実装者を同じ会話に入れるか、オーナーが残論点を裁定する。

## hollow-green 対策(工程5)

worker は skeleton 実装でもテストを green にできてしまう。「green」は代理指標で
あり、安価なモデルは代理指標を文字どおり最適化する(本番運用で繰り返し検出した)。
対策はすべて実戦済み:

- **テスト所有権の分離(contract-test-first)。** RED テストは別の強い lane が
  production パス上で「実際に落ちる」形で書いて所有する。実装者の仕事はそれを
  green にすることだけ。**実装者がテストに diff を入れたら即差し戻し。**
- **real-boundary green。** fake クライアント / 純 unit の green は工程3(PoC)の
  証拠にしかならない。工程5の merge gate は real 境界の harness(実 API サーバー
  等)での green を要求する。偽物の環境は正直な実装と空洞の実装を見分けられ
  ない — これは注意力の問題ではなく構造的な上限。
- **held-out テスト。** レビュー時、実装者が見たことのないテストを1本追加して
  実行する。skeleton は落ち、仕様の実装は通る。レビュー手順の1行で済む、
  最も安価で強い検出器。
- **同一 lane からの hollow-green 2回目は circuit breaker イベント。** 止めて
  方式を変える — contract-test-first への切り替え、台帳制のブースト、issue の
  切り直し。これは怠慢ではなく、scope 過大・gate 欠陥・ギア不足のシグナル。
- **gate は正直な green を許す構造であること。** テスト gate をレビューする
  とき(工程2)、正直な実装がその gate の環境で本当に green になれるかを
  確認する — 正直に通れない gate は捏造を誘発する。

## 進行管理ファイル

案件ごとに1本: 作業ディレクトリの `flow/<topic>-flow.md`。その案件のフロー
位置の唯一の正となる。

```markdown
# flow: <topic>
- started: <date> / owner: <lane or person>
- current stage: <N>. <name>

| stage | status | receipt (path + SHA) | notes |
|---|---|---|---|
(rows 0-8)

## Rollback history
- <date> stage N -> M: <one-line reason>
```

進む/戻るたびに必ず更新する。receipt 列は**パス + SHA**。
「口頭で done」は done ではない。

## 差し戻し早見表

| ズレが見つかった場所 | 戻る先 |
|---|---|
| 設計レビューで scout にない前提が見つかった | 1(scout 引き直し or 設計修正) |
| PoC で設計仮説が崩れた | 2 |
| 確証で未検証の段が残っていた | 3 |
| 実装が設計前提からズレた | 2(設計を改版する — その場のつじつま合わせ禁止) |
| 本番反映が失敗した(記録は残っている) | 4(packet 再発行・方式の再点検) |
| 観察で問題が見つかった | 裁定(2 か 5 か 6)— 自動ロールバック禁止 |

**gate を満たさないのに進みたくなったら**: 間違っているのは gate ではなく、
上流のどこかだ。差し戻し先を特定する。

## 誰が何をやるか(worker fleet を使う場合)

- 実装の重い工程(3〜5, 7)は **fleet 標準ギア**の worker へ(例: Codex fleet
  なら `gpt-5.6-terra` medium、Claude Code fleet なら **`sonnet`** が既定の
  実装者)。
- 上位ギア(Codex `sol` effort low、Claude Code `opus`)は「はまり突破」と
  blocking スポットレビュー専用で、**台帳制**(model-gear の台帳規約を参照)。
  常駐禁止。
- 工程6(live)は**人間の gate**: オーナー本人か明示的に授権された executor で
  あって、モデルではない。
- lane の生成と寿命は issue-lane スキル(1 issue = 1 lane)に従う。

## 停止条件(circuit breaker)

進捗は成果(実 mutation、readback、残工程)で数える。receipt の枚数や
successor の数では数えない。同一 operation の 2 rejection、30分の停滞、
protocol 欠陥3連続のいずれかに達したら: 止めて、次の試行を禁止し、方式自体を
設計し直す。fail-closed は壊れた方式を無限に繰り返す許可ではない。

## プロジェクトへの適応

プロジェクトに固有の正本フロー文書があればそちらが優先。本スキルは正本が
無いプロジェクトの既定値。採用するときは工程表をプロジェクトの正本文書に
コピーし、プロジェクト固有の gate・担当・パス規約をそこに蓄積していく。
トラック裁定(そのプロジェクトで何が消費財・顧客可視にあたるか)も同じ文書に
記録する。
